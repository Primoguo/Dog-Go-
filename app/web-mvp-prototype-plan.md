## 原型目标

先用移动端 Web 跑通自律狗 MVP 的核心体验：

领养狗狗 → 创建目标 → 首页狗狗小世界 → 完成打卡 → 狗狗反馈 → 查看进度 → 次日节奏恢复。

这个原型的目标不是完整商业产品，而是验证：

- 狗狗小世界是否比普通任务列表更有吸引力。
- 用户完成打卡后是否能感到狗狗在回应自己。
- 中断后的恢复任务是否让用户愿意回来。

## 技术取舍

MVP 原型建议先用本地移动端 Web，而不是直接做 iOS。

建议方案：

- 纯前端单页应用。
- 本地状态存储使用 `localStorage`。
- 首版不接真实账号系统。
- 首版不接后端。
- 首版不做真实推送。
- 先用可替换视觉占位资产，后续替换为正式插画。

推荐实现路径：

1. 先做静态移动端 Web 原型。
2. 再补交互状态和本地存储。
3. 最后替换狗狗和小院子视觉资产。

## MVP 页面范围

P0 页面：

| 页面 | 目的 |
| --- | --- |
| 领养页 | 选择首发 4 只狗之一 |
| 目标创建页 | 创建第一个自律目标 |
| 首页 | 狗狗小世界 + 今日目标 + 打卡 |
| 反馈页 | 完成后展示狗狗反馈和状态变化 |
| 进度页 | 展示亲密度、本周节奏、近 7 天 |
| 恢复状态 | 用户中断后展示小恢复任务 |

P1 页面或模块：

| 页面或模块 | 目的 |
| --- | --- |
| 打卡确认 | 备注和心情记录 |
| 登录保存提示 | 游客体验后的保存引导 |
| 目标降级模块 | 长期中断后调小目标 |
| 目标编辑 | 修改目标名称、频率、提醒 |

## 首版路由

建议用一个单页应用维护内部 view 状态：

| View | 说明 |
| --- | --- |
| `adopt` | 领养页 |
| `create-goal` | 目标创建页 |
| `home` | 首页 |
| `feedback` | 打卡反馈页 |
| `progress` | 进度页 |

状态跳转：

```text
adopt → create-goal → home
home → feedback → progress → home
home → progress → home
home(recovery) → feedback(recovery) → home
```

## 首页首版结构

```text
┌──────────────────────────────┐
│ 顶部问候                      │
├──────────────────────────────┤
│ 狗狗小世界                    │
│ 小院子 / 狗狗 / 道具 / 气泡    │
├──────────────────────────────┤
│ 今日目标卡片                  │
│ 主按钮                        │
│ 本周节奏 / 亲密度 / 进度入口  │
└──────────────────────────────┘
```

首版必须支持这些状态：

| 状态 | 首页表现 |
| --- | --- |
| 待完成 | 狗狗在小院子里等待，展示今日目标 |
| 已完成 | 狗狗 happy，按钮为“今天已完成” |
| 恢复任务 | 狗狗 waiting，展示更小任务 |
| 长期中断 | 展示调小目标建议 |

## 状态计算规则

完成主目标：

- 亲密度 +3。
- 健身目标：精力 +20，饱腹 +15。
- 学习目标：精力 +10，饱腹 +10。
- 作息目标：清洁 +20，精力 +15。
- 今日状态变为已完成。

完成恢复任务：

- 亲密度 +2。
- 饱腹 +10。
- 清洁 +10。
- 精力 +10。
- 节奏状态变为恢复中。

漏 1 天：

- 不扣亲密度。
- 不降等级。
- 狗狗姿态变为 `waiting`。
- 今日主任务替换为恢复任务。

连续 3 天：

- 亲密度 +8。
- 展示节点反馈。

连续 7 天：

- 亲密度 +20。
- 展示节点反馈。

## 本地存储

建议存储 key：

```text
zilvgou.appState.v1
```

存储内容：

- 用户是否完成领养。
- 已选择狗狗。
- 当前目标。
- 狗狗状态。
- 打卡历史。
- 当前 view。

重置方式：

- 原型阶段提供一个隐藏或开发按钮“重置体验”。

## 首版视觉资产策略

在正式插画出来前，Web 原型可以先使用结构化占位：

- 小院子背景：用简单色块和布局表达层级。
- 狗狗：用临时占位图位或后续生成插画替换。
- 道具：先用简单标签和图形位置表达。

注意：

- 占位必须服务布局验证，不把它当最终视觉。
- 正式评审前需要替换为真实插画或生成视觉资产。
- 不把狗狗做成孤立头像，必须放在小世界里。

## 第一轮开发任务

### Task 1：搭应用外壳

- 创建移动端 Web 原型目录。
- 建立 HTML/CSS/JS 基础结构。
- 实现 view 切换。

验收：

- 能在浏览器打开。
- 默认进入领养页。
- 页面宽度按移动端设计。

### Task 2：领养页

- 展示柴犬、金毛、边牧、中华田园犬。
- 展示性格标签和领养预览文案。
- 选择后进入目标创建页。

验收：

- 狗狗选择不绑定任务类型。
- 选择结果保存到本地状态。

### Task 3：目标创建页

- 支持健身、学习、作息模板。
- 默认推荐健身。
- 创建后进入首页。

验收：

- 目标名称、类型、频率被保存。
- 首页能展示当前目标。

### Task 4：首页狗狗小世界

- 展示顶部问候。
- 展示狗狗小世界。
- 根据目标类型展示道具。
- 展示对话气泡。
- 展示今日目标和主按钮。

验收：

- 首页第一眼是狗狗小世界。
- 打卡按钮首屏可见。

### Task 5：完成打卡和反馈

- 点击完成今天。
- 更新狗狗状态。
- 展示反馈页。
- 更新亲密度和生活状态。

验收：

- 同一天重复完成不重复加分。
- 反馈文案符合所选狗狗语气。

### Task 6：进度页

- 展示亲密度、等级。
- 展示本周节奏。
- 展示近 7 天。
- 展示心情、饱腹、清洁、精力。

验收：

- 数据来自本地状态。
- 不做复杂图表。

### Task 7：恢复状态

- 模拟或判断昨日未完成。
- 首页展示恢复任务。
- 完成恢复任务后进入反馈。

验收：

- 不出现失败、清零、惩罚文案。
- 亲密度不下降。

## 原型验收标准

- 5 分钟内能从领养走到首次打卡反馈。
- 首页核心视觉是狗狗小世界，不是任务列表。
- 用户能清楚知道今天该点哪里。
- 不同狗狗说话语气有差异。
- 不同目标类型有不同道具呼应。
- 中断恢复状态不责备用户。
- 刷新后状态仍保留。

## 下一步

建议下一步直接创建移动端 Web 原型：

```text
app/web-mvp/
```

首版可以用无构建工具的静态 HTML/CSS/JS 实现，降低启动成本。等核心体验跑通后，再决定是否迁移到 React、SwiftUI 或其他正式技术栈。
