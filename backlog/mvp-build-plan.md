# 自律狗 MVP 第一版开发顺序

来源：[MVP PRD](../docs/zilvgou-mvp-prd.md) 与 [用户故事 Backlog](./mvp-user-stories.md)

## 目标

用最短路径做出可测试的 iOS MVP 闭环：

领养狗狗 → 创建健身目标 → 今日打卡 → 狗狗反馈 → 进度变化 → 次日或断签后的节奏恢复。

首版成功不以功能完整度判断，而以用户是否愿意连续回来判断。

## 推荐里程碑

### Milestone 0：产品与设计定稿

目标：让工程可以开工，避免在核心规则上反复摇摆。

输出物：

- 核心用户流程图：首页狗狗状态 → 今日目标 → 打卡 → 狗狗反馈 → 进度页 → 节奏恢复。
- 首发狗狗名单：建议柴犬、金毛、边牧、中华田园犬。
- 每只狗狗的性格标签和 5-8 条反馈文案。
- 目标模板文案：健身、学习、作息三类。
- 状态规则 v1：亲密度、等级、心情、饱腹、清洁、精力。
- 一天结算规则：建议本地时区 + 凌晨 4:00 归属前一天。

完成标准：

- 设计和工程对 P0 流程无阻塞问题。
- P0 文案足够覆盖普通完成、连续节点、恢复任务三类场景。

### Milestone 1：本地可跑的核心闭环

目标：不依赖登录和后端，先跑通产品感觉。

范围：

- 领养狗狗。
- 创建一个默认健身目标。
- 今日任务展示。
- 一键完成打卡。
- 打卡后狗狗反馈。
- 本地保存狗狗、目标和当天打卡状态。

建议任务：

1. 建立基础应用结构和导航。
2. 建立本地数据模型：
   - DogProfile
   - Goal
   - CheckIn
   - DogState
   - RhythmStatus
3. 实现领养页。
4. 实现目标创建页，默认推荐健身。
5. 实现首页今日任务。
6. 实现一键打卡和重复点击保护。
7. 实现打卡反馈页。

完成标准：

- 新用户可以在 2 分钟内完成领养、创建目标、首次打卡。
- 首次打卡后能看到亲密度和状态变化。
- 关闭再打开应用后，体验数据仍在本地保留。

### Milestone 2：成长与进度

目标：让打卡不只是“完成任务”，而是能看到关系和节奏变化。

范围：

- 亲密度增长。
- 等级计算。
- 心情、饱腹、清洁、精力状态更新。
- 本周完成率。
- 近 7 天节奏展示。
- 连续 3 天、7 天节点反馈。

建议任务：

1. 实现状态计算服务。
2. 实现亲密度和等级规则。
3. 实现近 7 天打卡聚合。
4. 实现进度页。
5. 实现连续 3 天、7 天奖励。
6. 增加状态上下限保护。

完成标准：

- 完成主目标增加亲密度，但未完成不扣亲密度。
- 本周完成率和近 7 天状态显示正确。
- 连续节点能触发额外反馈和奖励。

### Milestone 3：节奏恢复

目标：验证“断了也能回来”的核心差异化。

范围：

- 昨日未完成识别。
- 恢复任务生成。
- 恢复任务完成。
- 长期中断降级建议。

建议任务：

1. 实现昨日未完成判断。
2. 实现恢复任务模板。
3. 首页展示恢复提示。
4. 完成恢复任务后更新状态和反馈。
5. 连续 3 天未完成时展示降低目标难度建议。

完成标准：

- 漏掉 1 天后，首页不展示失败惩罚，而展示小恢复任务。
- 完成恢复任务后，亲密度小幅增加。
- 连续 3 天未完成后，用户可以降低目标难度或重新设定。

### Milestone 4：游客体验到登录保存

目标：让体验不被登录打断，同时正式进度可保存。

范围：

- 游客体验状态。
- 登录提示。
- 登录后数据迁移。
- 正式数据保存。

建议任务：

1. 标记用户状态：guest 或 authenticated。
2. 在保存长期进度、查看历史、离开体验关键节点时提示登录。
3. 实现登录流程占位或真实登录方案。
4. 登录后迁移本地狗狗和目标。
5. 处理迁移失败提示。

完成标准：

- 未登录用户可完整体验一次核心闭环。
- 登录后保留本次体验狗狗和第一个目标。
- 用户不会因为登录丢失体验进度。

### Milestone 5：埋点与测试版准备

目标：让首批测试能回答 MVP 是否成立。

范围：

- 核心事件埋点。
- 基础错误处理。
- 测试用户任务脚本。
- App Store / TestFlight 准备项。

建议事件：

| 事件 | 目的 |
| --- | --- |
| guest_experience_started | 体验入口转化 |
| dog_selected | 狗狗吸引力 |
| goal_created | 目标创建完成率 |
| first_checkin_completed | 首次打卡完成率 |
| dog_feedback_viewed | 反馈触达 |
| progress_viewed | 成长系统查看 |
| recovery_prompt_viewed | 断签恢复触达 |
| recovery_task_completed | 恢复机制有效性 |
| login_prompt_viewed | 登录提示触达 |
| signup_completed | 游客转登录 |

完成标准：

- 可计算 D1、D7 回访。
- 可计算目标创建完成率、首次打卡完成率、节奏恢复完成率。
- 可给 8-12 位测试用户发放明确测试任务。

## P0 范围

必须包含：

- 领养狗狗。
- 游客体验。
- 创建健身目标。
- 今日任务。
- 一键打卡。
- 狗狗反馈。
- 亲密度和等级。
- 本周完成率或近 7 天节奏。
- 节奏恢复。
- 登录保存或登录保存占位方案。
- 基础埋点。

## P1 范围

有余力再做：

- 编辑目标。
- 暂停目标。
- 学习和作息目标完整体验。
- 更多狗狗状态展示。
- 长期中断后的目标降级。
- 更丰富的反馈文案。

## P2 范围

MVP 后再做：

- 5 种以上狗狗。
- 皮肤、装扮和商城。
- 复杂趋势分析。
- 社交监督。
- AI 长对话。
- 多平台同步。

## 建议开发切片

第一周：

- 完成 Milestone 0。
- 完成本地数据模型。
- 完成领养、目标创建、今日任务静态流程。

第二周：

- 完成一键打卡、反馈页、本地保存。
- 完成亲密度、等级和基础状态计算。
- 完成进度页。

第三周：

- 完成节奏恢复。
- 完成游客到登录保存。
- 接入基础埋点。
- 准备首批测试。

## 风险优先级

最高风险：

- 狗狗陪伴是否真的提升回访。
- 断签恢复是否让用户愿意回来。
- 游客到登录是否会造成进度丢失感。

中等风险：

- 状态系统过复杂，用户看不懂。
- 目标模板过泛，健身首发不够聚焦。
- 文案太幼稚或太说教。

降低风险的做法：

- 首屏只突出心情、亲密度和今日目标。
- 先把健身闭环做扎实，再补学习和作息。
- 用 8-12 位目标用户做 7 天测试。
- 重点观察断签后的次日回访，而不是只看首次打卡。

## 首批测试任务

给测试用户的任务建议：

1. 第一天不登录，完成领养、创建目标和首次打卡。
2. 第二天回来完成一次正常打卡。
3. 第三天故意不打卡或自然中断。
4. 第四天观察是否愿意做恢复任务。
5. 第七天访谈：是否因为狗狗反馈想回来，还是只把它当普通打卡工具。

核心访谈问题：

- 你会怎么描述这只狗和你的关系？
- 哪一句反馈让你想继续，哪一句让你无感？
- 中断后的提示有没有让你更愿意回来？
- 你希望它鼓励你、提醒你、监督你，还是安静陪你？
- 如果只能留下一个状态，你最在意亲密度、心情、等级还是连续节奏？
